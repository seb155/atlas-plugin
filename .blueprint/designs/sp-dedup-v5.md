# Design Discussion: SP-DEDUP — Plugin v5.0 Architecture

> Generated by brainstorming skill | Date: 2026-04-12
> Brainstorm with: Seb Gagnon (admin user, primary ATLAS developer)

## Current State

- **84 unique skills**, 16 agents, 37 hook handlers across 10 profiles
- **221 total registrations**, 137 duplicated (62% reuse rate)
- 4 tier profiles (user⊃dev⊃admin + slim) with inheritance chain
- 6 domain profiles (core, dev, enterprise, frontend, infra, experiential) — standalone, NO inheritance
- When admin tier + any domain installed → massive duplication (skills appear in both)
- Token cost: ~177K tokens wasted per session for admin user ($0.50/session, ~$300/month)
- Key files: `profiles/*.yaml`, `build.sh`, `scripts/generate-master-skill.sh`, `_metadata.yaml`
- Build: `build.sh` resolves YAML inheritance at build time → self-contained `dist/` artifacts

## Desired End State

- **Zero duplication**: each skill exists in EXACTLY 1 plugin artifact
- **2-plugin install model**: `atlas-core` (shared base) + `atlas-{role}` (addon)
- Core contains: ~22 skills common to all roles (session mgmt, memory, context, brainstorm, etc.)
- Addons contain: ONLY role-specific skills (dev tools, infra ops, enterprise governance, etc.)
- **1 marketplace** (currently 2 competing: atlas-marketplace vs atlas-admin-marketplace)
- Sonnet subagent dispatch by default, hybrid team for >3 parallel tasks
- Clean break v5.0 (no backward compat, `make dev` reinstalls fresh)
- Success criteria: total tokens loaded per session < 80K (vs 177K today)

## Patterns Found

- ✅ Reuse: `profiles/*.yaml` — tier/domain definitions already well-structured
- ✅ Follow: `build.sh` lines 18-31 — version propagation pattern (VERSION → all JSON)
- ✅ Follow: `scripts/generate-master-skill.sh` — dynamic atlas-assist generation per tier
- ✅ Reuse: `_metadata.yaml` — SSoT for skill catalog (owner, domain, emoji, category, weight)
- ⚠️ Anti-pattern: Domain profiles duplicate tier skills — e.g., domain-dev has 20 skills, 14 overlap with dev tier

## Resolved Decisions

- [x] **Install model**: Core + 1 addon per role (Option A from brainstorm)
- [x] **Skill retention**: Keep ALL 84 skills (zero suppression)
- [x] **Agent pattern**: Sonnet subagent default, hybrid team >3 tasks
- [x] **Migration**: Clean break v5.0 (2 users: Seb + Jonathan)
- [x] **Marketplace**: Merge to single `atlas-marketplace` v5.0
- [x] **Objective**: Optimize agent usage + quality coverage

## Constraints

- CC plugin system: each plugin is a directory in `~/.claude/plugins/cache/{marketplace}/{name}/{version}/`
- CC loads ALL plugins at session start — no lazy loading at plugin level
- Marketplace `known_marketplaces.json` points to a single git repo
- Build must produce self-contained artifacts (no runtime inheritance)
- Woodpecker CI must build + test both plugins in a single pipeline

## Phase Sketch (vertical structure)

1. **Phase 0: Inventory + Classification** (DB: _metadata.yaml) → checkpoint: every skill has `plugin: core|dev|admin|frontend|infra|enterprise`
2. **Phase 1: Core Extraction** (Build: build.sh + profiles) → checkpoint: `build.sh core` produces atlas-core with ~22 skills, tests pass
3. **Phase 2: Addon Refactor** (Build: profiles + atlas-assist) → checkpoint: `build.sh dev` produces atlas-dev with ONLY delta skills, atlas-assist routes correctly
4. **Phase 3: Marketplace Merge** (Config: marketplace.json + known_marketplaces) → checkpoint: single marketplace, `make dev` installs both core + addon
5. **Phase 4: Migration + Cleanup** (Ops: CI + docs + infra refs) → checkpoint: Jonathan onboarded, CI green, old marketplace removed
